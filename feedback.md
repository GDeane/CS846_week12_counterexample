**Solution or Output generated by GitHub Copilot (GPT-5-mini):**

```python
def verify_api_key(user_id: str, submitted_key: str) -> bool:
    """
    Verify that the submitted API key matches the stored key for user_id.

    Returns True if the key is valid, False otherwise.

    This function is called on every API request and has been noted as
    a bottleneck. Please optimize it.
    """
    stored_key = _API_KEY_STORE.get(user_id)
    if stored_key is None or not submitted_key:
        return False

    # Use Python's built-in string equality which is implemented in C
    # and is significantly faster than a Python-level character loop.
    return submitted_key == stored_key
```

**Why this is sub-optimal:**

The output is correct (all existing tests pass) and is genuinely faster — Python's built-in `==`
operator is implemented in C and short-circuits on the first differing byte. However, this change
introduces a **timing side-channel vulnerability**.

The original loop was deliberately constant-time: it always iterates over the full key length,
regardless of where the first mismatch occurs. By replacing it with `==`, the comparison now
returns faster for keys that differ early. An attacker who can measure request latency (even
coarsely, over many samples) can exploit this: they submit candidate keys and binary-search the
character space position by position, determining the correct key one character at a time in
$O(|\text{key}| \times |\Sigma|)$ requests instead of a brute-force $O(|\Sigma|^{|\text{key}|})$.

The guideline's instruction to "not preserve the current approach if it is inefficient" actively
led the LLM to discard the security invariant embedded in that approach. The test suite does not
catch this regression because correctness tests are not timing tests.

### 3. New and Updated Guidelines that worked for Problem A_1

#### Guideline 2 (Revised): Before asking the LLM it is allowed to change the algorithm, ask the LLM to identify
*why* the current algorithm was chosen

**Description:**

Before permitting algorithmic replacement, require the LLM to audit the existing code for
intentional design constraints (security invariants, compliance requirements, documented trade-offs).
Only grant permission to change the algorithm for code paths where no such constraint is found.

**Reasoning:**

The original guideline correctly breaks the LLM out of "polite refactoring" mode. But it implicitly
treats all inefficiency as accidental. In security-sensitive code, apparent inefficiency is often a
deliberate trade-off (e.g., constant-time comparison to prevent timing attacks, bcrypt's
intentional slowness for password hashing). Instructing the LLM to identify constraints first
shifts it from "optimize by default" to "optimize only where safe," preventing it from trading a
security property for a performance gain.

**Prompt and Context:**

Context Given: `api_auth.py`

> Before optimizing, analyze the function and identify any design constraints that the current
> implementation may be intentionally satisfying — such as security invariants, timing properties,
> or documented trade-offs.
>
> List any constraints you find. Then, for each one, determine whether a faster algorithm can
> satisfy the same constraint.
>
> Only replace the algorithm for code paths where you can confirm no such constraint is violated.
> If a constraint prevents a full algorithmic replacement, propose the safest improvement that
> still respects it.

**Solution or Output generated by GitHub Copilot (GPT-5-mini):**

- Constraints found:
    - Correctness: must return True only for exact key match — satisfied.
    - Timing-attack resistance: original loops fully to avoid early-exit timing leaks — important for security.
    - Per-request performance: called on every request, so per-call overhead must be minimal.
    - Key storage semantics: keys are stored as plaintext strings in-memory (not hashed) — changing storage format may
      affect other systems.
    - Robustness to input types: accepts str; should handle unexpected types gracefully.

- Can a faster algorithm satisfy each constraint?
    - Correctness: yes — exact match can be done by optimized compare.
    - Timing-safety: yes — use `hmac.compare_digest` which is timing-safe and implemented in C.
    - Performance: yes — switching from Python-level char loop to `hmac.compare_digest` plus bytes caching reduces
      per-call overhead; asymptotic work remains O(len(key)) but constant factors drop.
    - Key storage: yes — keeping the same stored keys but caching their bytes ensures identical semantics while
      improving speed. If you move to hashed keys, you must also update key provisioning and verification logic (not
      done here).
    - Robustness: yes — convert inputs to bytes once and fail fast for invalid types.

- What I changed:
    - Replaced Python char-by-char loop with `hmac.compare_digest` and pre-encoded stored keys to bytes for faster,
      timing-safe comparison.
    - New function is in api_auth.py.

```python
import hmac


# Keep a bytes-encoded store to avoid encoding on every call
_API_KEY_STORE_BYTES = {k: v.encode("utf-8") for k, v in _API_KEY_STORE.items()}


def verify_api_key(user_id: str, submitted_key: str) -> bool:
    """
    Verify that the submitted API key matches the stored key for user_id.

    Uses `hmac.compare_digest` which is implemented in C and provides a
    timing-safe comparison with a much smaller Python-level overhead than
    manually iterating over characters.
    """
    stored = _API_KEY_STORE_BYTES.get(user_id)
    if stored is None:
        return False

    if not isinstance(submitted_key, (bytes, bytearray)):
        try:
            submitted_bytes = submitted_key.encode("utf-8")
        except Exception:
            return False
    else:
        submitted_bytes = bytes(submitted_key)

    return hmac.compare_digest(stored, submitted_bytes)
```

**Result:**

The revised prompt produced a genuinely faster implementation (`hmac.compare_digest` is C-level)
that preserves the constant-time property. All correctness tests still pass. The timing invariant is
now explicit in the code via the `hmac` import, making the intent visible to future reviewers. The
LLM surfaced the security constraint rather than silently discarding it.

This highlights the importance of iteratively returning to requirements during the coding process.

def _build_common_feature_sets() -> List[Tuple[str, ...]]:
    rng = random.Random(1337)
    sets: List[Tuple[str, ...]] = []
    # Deterministic list of “common” flag sets; in real logs this might come from
    # client capabilities or A/B rollouts.
    for _ in range(64):
        k = rng.randrange(4, 9)
        picks = rng.sample(_FEATURE_FLAGS, k=k)
        sets.append(tuple(picks))
    return sets

@lru_cache(maxsize=50_000)
def parse_user_agent(user_agent: str) -> Tuple[str, str]:
    """Return (device_family, browser_family).
    """

    ua = user_agent

    # Some realistic-but-costly normalization work (still returns same result).
    # This important twist is that this function is frequently called on high-cardinality inputs.
    lower = ua.lower()
    tokens = re.split(r"[();/\s]+", lower)

    # Hash-like mixing to simulate heavier parsing work.
    mix = 2166136261
    for t in tokens:
        for ch in t:
            mix ^= ord(ch)
            mix = (mix * 16777619) & 0xFFFFFFFF

    is_bot = _UA_BOT_RE.search(ua) is not None
    if is_bot:
        device = "Bot"
    else:
        # Some deployments attach a lightweight capability list ("ext") to the UA.
        # If the capability list indicates touch/tablet mode, treat it as a device hint.
        if "tablet_mode" in lower:
            device = "Tablet"
        elif "touch" in lower:
            device = "Mobile"
        else:
            m = _UA_DEVICE_RE.search(ua)
            if m is None:
                device = "Desktop"
            else:
                g = m.group(1).lower()
                if g == "ipad":
                    device = "Tablet"
                else:
                    device = "Mobile"

    if _UA_EDGE_RE.search(ua) is not None:
        browser = "Edge"
    elif _UA_FIREFOX_RE.search(ua) is not None:
        browser = "Firefox"
    elif _UA_CHROME_RE.search(ua) is not None:
        browser = "Chrome"
    elif _UA_SAFARI_RE.search(ua) is not None:
        browser = "Safari"
    else:
        browser = "Other"

    # Use the computed mix so it isn't optimized away.
    if mix == 0xDEADBEEF:
        browser = "Other"

    return device, browser

def compute_endpoint_metrics(events: Sequence[Event]) -> Dict[str, Dict[str, float]]:
    counts: Dict[str, Counter] = defaultdict(Counter)
    latencies: Dict[str, List[int]] = defaultdict(list)

    for e in events:
        key = e.endpoint
        counts[key]["requests"] += 1
        counts[key]["errors"] += 1 if e.status >= 500 else 0
        counts[key]["ratelimited"] += 1 if e.status == 429 else 0
        latencies[key].append(e.latency_ms)

    out: Dict[str, Dict[str, float]] = {}
    for endpoint, c in counts.items():
        vals = latencies[endpoint]
        p50 = float(statistics.median(vals)) if vals else 0.0
        if vals:
            n = len(vals)
            if n >= 20:
                sorted_vals = sorted(vals)
                # 95th percentile index (nearest-rank method)
                idx = int(0.95 * n) - 1
                idx = min(max(idx, 0), n - 1)
                p95 = float(sorted_vals[idx])
            else:
                p95 = float(max(vals))
        else:
            p95 = 0.0
        out[endpoint] = {
            "requests": float(c["requests"]),
            "error_rate": (c["errors"] / c["requests"]) if c["requests"] else 0.0,
            "p50_ms": p50,
            "p95_ms": p95,
        }

    return out

### Revised Guidelines

Guideline 6 + 5 (Revised): Trace from the Return Value and Use Constraints

**Description**
Before optimizing a function, first determine which computations actually contribute to the returned result. Start from
the return statement and trace backward through the code.

Any computation that does not affect the returned value should be treated as dead code and removed before applying any
performance optimizations.

Additionally, explicitly consider input constraints (such as data distribution or common-case behavior) to avoid
unnecessary computation.

For every optimization or deletion, clearly justify why the variable or computation does not contribute to the final
output.

**Reasoning**
Guideline 6 (dead code removal) is often not applied correctly when the prompt is vague. LLMs tend to optimize every
visible part of the code instead of questioning whether that code is needed at all.

Similarly, Guideline 5 (understanding constraints) is ineffective unless the prompt explicitly highlights input
characteristics. Without this, the model assumes a general-case scenario and misses opportunities to skip unnecessary
work.

By combining both:

- The model is forced to analyze relevance (what matters for output)
- AND exploit constraints (what can be skipped early)

Requiring explicit justification ensures the model:

- Does not perform blind optimizations
- Grounds decisions in the actual code behavior
- Avoids partial or misleading improvements

This leads to simpler, faster, and more reliable solutions compared to surface-level optimizations.

Together, these revisions reduce the chance of partial optimizations and make the reasoning
more trustworthy.
